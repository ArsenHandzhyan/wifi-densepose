# Анализ модели V46 Drift-Robust CSI

**Дата:** 2026-04-01
**Модель:** GradientBoostingClassifier (GBM-500)
**CV F1-macro:** 0.9850
**Классы:** empty / static / motion

---

## 1. Архитектура фичей V46: 11 drift-resistant признаков

V46 использует **77 признаков** (11 на каждый из 7 узлов ESP32). Ключевое отличие от V45 --
все признаки **нормализованы** и устойчивы к дрейфу амплитуд (температура, влажность, время суток).

### Полный список признаков на узел:

| # | Признак | Формула | Назначение |
|---|---------|---------|------------|
| 0 | `std_rssi` | std(RSSI) | Вариативность RSSI -- индикатор движения |
| 1 | `cv_amp` | std(amp)/mean(amp) | Коэффициент вариации -- нормализованный разброс |
| 2 | `amp_range_ratio` | (max-min)/(mean+1) | Нормализованный динамический диапазон |
| 3 | `temporal_var` | mean(std по поднесущим)/mean(amp) | Нормализованная временная дисперсия -- сигнатура движения |
| 4 | `low_ratio` | mean(low_band)/(mean+1) | Спектральная форма -- нижняя треть |
| 5 | `mid_ratio` | mean(mid_band)/(mean+1) | Спектральная форма -- средняя треть |
| 6 | `high_ratio` | mean(high_band)/(mean+1) | Спектральная форма -- верхняя треть |
| 7 | `pkt_std_amp` | std(mean_per_packet)/mean | CV временной энергии |
| 8 | `sc_entropy` | -sum(p*log(p)) | Спектральная сложность (энтропия поднесущих) |
| 9 | `rssi_range` | max(RSSI)-min(RSSI) | Размах RSSI в окне -- индикатор перемещения |
| 10 | `amp_iqr_ratio` | (Q75-Q25)/(median+1) | Робастная мера дисперсии |

**Принцип drift-устойчивости:** ни один признак не использует абсолютные значения амплитуд напрямую. Все метрики -- это отношения, коэффициенты вариации или разности. Если базовая амплитуда дрейфует на 40--70% между записями (как обнаружено в V45), соотношения остаются стабильными.

## 3. Анализ решающих границ

Анализ первых деревьев GBM-500 показывает чёткие пороги для каждого класса:

### Класс EMPTY
- **Основной предикат:** `n06_temporal_var <= 0.14`
- Логика: если временная дисперсия на N06 ниже порога -- нет движения и нет тела, значит пусто
- Порог стабильно повторяется в деревьях 0-4 (0.1400--0.1425)

### Класс MOTION
- **Основной предикат:** `n07_rssi_range > 2.5` ИЛИ `n01_std_rssi > 0.47`
- Логика: движение создает колебания RSSI на нескольких узлах
- Вторичный split: `n01_rssi_range > 1.5`

### Класс STATIC
- **Основной предикат:** `n03_amp_iqr_ratio > 0.985`
- Логика: стоящий человек создает характерный паттерн дисперсии амплитуд, отличный от пустого помещения
- Порог стабильный (0.9854--0.9855 в деревьях 0-4)
- Альтернативный split: `n04_amp_iqr_ratio > 0.84`

### Иерархия решений:
```
temporal_var(N06) > 0.14?
  YES -> rssi_range(N07) > 2.5?
    YES -> MOTION
    NO  -> amp_iqr_ratio(N03) > 0.985?
      YES -> STATIC
      NO  -> дополнительные деревья...
  NO  -> EMPTY
```

## 4. Сравнение V45 vs V46: почему V45 провалилась на live-данных

### V45 -- признаки (8 на узел, 56 всего)
| Признак | Sum Importance | **Проблема** |
|---------|---------------|-------------|
| `std_rssi` | **0.5508** | OK, drift-free |
| `high_amp` | 0.1899 | **АБСОЛЮТНЫЕ амплитуды -- дрейфуют!** |
| `max_amp` | 0.0807 | **АБСОЛЮТНЫЕ -- дрейфуют!** |
| `mean_amp` | 0.0759 | **АБСОЛЮТНЫЕ -- дрейфуют!** |
| `mean_rssi` | 0.0400 | **АБСОЛЮТНЫЙ RSSI -- дрейфует!** |
| `std_amp` | 0.0273 | Зависит от масштаба |
| `mid_amp` | 0.0191 | **АБСОЛЮТНЫЕ** |
| `low_amp` | 0.0163 | **АБСОЛЮТНЫЕ** |

### Корневая причина отказа V45

1. **55% importance на std_rssi** -- этот признак drift-free, но сам по себе недостаточен для 3-class разделения
2. **35% importance на абсолютных амплитудах** (high_amp, max_amp, mean_amp) -- эти признаки дрейфуют на 40-70% между записями из-за:
   - Температуры в гараже (ночь vs день)
   - Влажности
   - Временных сдвигов базовой линии ESP32
3. **CV-score V45 = 0.9825 vs V46 = 0.9850** -- на кросс-валидации разница минимальная (0.25%), потому что тренировочные данные записаны в одних условиях. На live-данных с другими условиями V45 ломается.
4. **V46 добавил empty_diag_v45_live** -- запись пустого гаража в "дрейфовых" условиях, чтобы обучить модель на сдвинутом распределении.

### Ключевое отличие V46:
- **0% importance на абсолютных амплитудах** (таких признаков просто нет)
- **Все 77 признаков -- отношения или разности**, инвариантные к масштабу
- temporal_var нормализован на mean_amp, поэтому drift отменяется

### Проблемный клип: `walk_center_slow_1`

- **Все 2 окна классифицированы как STATIC** вместо MOTION
- Причина: очень медленная ходьба в центре зоны генерирует minimal temporal variance
- При 7-пакетном окне (WINDOW_SIZE=7) и медленной ходьбе (~10 пакетов в секунду, ~0.7 сек на окно), перемещение составляет <20 см

### Почему walk_center_slow_2 работает, а slow_1 -- нет?
- slow_1 имеет всего 2 окна (меньше данных, мало пакетов от ноды)
- Вероятно, человек стоял/двигался минимально в начале записи
- slow_2 (3 окна) -- чуть больше перемещение, достаточное для превышения порога

### Общая точность на walking: **13/14 = 92.9%** (по клипам), **39/41 = 95.1%** (по окнам)

### Рекомендации по walking-данным:
1. Записать больше slow walking -- минимум 5 сессий по 2-3 минуты
2. Включить "shuffling" (переминание на месте) -- пограничный случай static/motion
3. Записать walking с остановками (walk-stop-walk) для тренировки переходов

## 6. Узкое место: класс STATIC (26 сэмплов)

### Текущее состояние
- **26 сэмплов static** после балансировки (undersampling target = 26*2.5 = 65)
- **65 empty, 65 motion** -- обрезаны от гораздо большего числа
- Всего 156 сэмплов в обучении

### Парадокс: 16390 доступных окон, но только 26 в модели

Причина -- **undersampling со stride=1.** Скрипт V46 вызывает `load_class_data` с stride=1 по умолчанию. Но при WINDOW_SIZE=7 и сотнях пакетов, один файл marker генерирует сотни перекрывающихся окон. Значит, **raw static windows = 16390**, но undersampling обрезал до 26.

Проблема: **26 -- это min_class_count** перед балансировкой. Это значит, что один из классов (static) имел всего 26 окон ДО балансировки. Вероятно, проблема в stride или MIN_NODES filter.

### Реальная диагностика

Из meta: `static_files=38`, но windows=26. Значит из 38 файлов (16390 потенциальных окон) только 26 прошли через `extract_features_v46` с active >= MIN_NODES(5). **Это маловероятно** -- анализ показал, что все файлы имеют act=7.

Более вероятная причина: balancing обрезает static до target_per_class = min_class * 2.5, но если **motion изначально имел ~26 окон** (14 walking файлов по ~2-4 окна = ~40), а static тоже имел мало окон с stride=1 в коротких recording...

**Уточнение:** walking файлы дают 2-4 окна каждый (всего ~41), а static marker файлы -- сотни. Значит min_class = motion (~41), target = 102. Но meta показывает static=26 -- значит, V46 loading использует stride, ограничивающий число окон.

### Как собрать больше static-данных

1. **Быстрый способ (без новых записей):** Увеличить stride для static/empty классов до 3-5, а для motion оставить 1. Это даст больше уникальных окон из существующих записей.

2. **Средний способ:** Записать 5-10 минутных сессий стояния в разных позициях (центр, дверь, проход) в РАЗНОЕ время суток для захвата drift-условий:
   - Утро (холодный гараж)
   - День (прогретый)
   - Ночь (baseline)

3. **Идеальный способ:** Записать static в дрейфовых условиях:
   ```
   # Протокол записи static (30 мин):
   - 5 мин: стояние у маркера 1 (тихо)
   - 5 мин: стояние у маркера 4 (центр)
   - 5 мин: стояние у двери
   - 5 мин: сидение на стуле (center)
   - 5 мин: стояние у маркера 7
   - 5 мин: стояние у маркера 8
   ```

4. **Augmentation:** Из каждой 1-минутной записи маркера можно извлечь ~500+ окон (stride=1). Для 18 static-файлов это >9000 уникальных окон. **Нужно убрать ограничение, которое обрезает static до 26.**

## 8. Рекомендации (приоритет)

### P0 -- Критические

1. **Исправить undersampling static.** 26 сэмплов -- опасно мало. При 16390 доступных окнах нужно загружать хотя бы 200-500 (stride=10-20 для длинных записей).

2. **Добавить augmentation/oversampling для static.** SMOTE или случайный oversampling до уровня empty/motion (65+).

3. **Записать static в drift-условиях.** Минимум 3 сессии в разное время суток (как сделано для empty с empty_diag_v45_live).

### P1 -- Важные

4. **Добавить slow walking данные.** walk_center_slow_1 misclassified -- нужно 5+ сессий медленной ходьбы.

5. **Добавить переходные сценарии.** walk-stop-walk, enter-stand-exit для тренировки границы motion/static.

6. **Мониторинг N06 и N03.** Эти два узла дают 56.6% importance. Нужен алерт при потере связи с ними.

### P2 -- Улучшения

7. **Feature pruning.** low_ratio (0.74%), pkt_std_amp (0.60%), cv_amp (1.59%) -- кандидаты на удаление. Сокращение до 8 признаков на узел (56 total) без потери качества.

8. **Увеличить WINDOW_SIZE до 10-14** для медленной ходьбы (больше temporal context).

9. **Добавить inter-node ratio features.** Например, temporal_var(N06)/temporal_var(N03) -- соотношение между узлами даст ещё один уровень drift-protection.

10. **Holdout test set.** Текущая оценка -- только CV на 156 сэмплах. Нужен отдельный holdout из записей другого дня/времени.

## 9. Заключение

V46 -- существенный шаг вперёд по сравнению с V45. Замена абсолютных амплитуд на нормализованные отношения устранила 40-70% drift, который ломал V45 на live-данных. Модель корректно классифицирует 95%+ walking окон и имеет CV F1=0.985.

Главная уязвимость -- **26 static сэмплов.** Это критический bottleneck, который нужно устранить в первую очередь через увеличение stride при загрузке и/или сбор дополнительных static-записей в дрейфовых условиях.

Вторая уязвимость -- **зависимость от N06** (34.2% importance). Потеря этого узла критически ослабит модель. Рекомендуется добавить fallback-режим с перетренировкой на 6 узлах.
